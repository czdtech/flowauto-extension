# FlowAuto 图生图/视频批处理阶段进展总结（2026-03-10）

## 1. 目标与范围

本阶段围绕 `FlowAuto` 在 Google Flow 场景下的「参考图注入 + 批量生图/生视频」能力进行真实可用性修复，重点目标：

- 跑通从侧边栏导入资源到 Flow 页面生成的端到端链路；
- 解决参考图“未上传/未生效/错配/复用效率低”的核心问题；
- 让多提示词与多参考图场景（含 M×N）可稳定执行；
- 基于真实浏览器交互与日志（Console/HAR）持续回归，而非仅静态代码推断。

---

## 2. 关键问题演进与处理结果

### 2.1 上传入口交互不稳定（早期）

**现象**
- 插件侧“目录导入”显示空白，用户无法看到已选文件；
- 点击上传后出现浏览器行为异常，且控制台出现：
  `File chooser dialog can only be shown with a user activation.`

**根因**
- 对文件输入框触发链路的拦截时机和方式不稳；
- 早期方案对 Flow 内部上传机制的适配不够精确。

**处理**
- 将上传注入策略调整为“更安全的实例级 `click()` 拦截 + MutationObserver 监听”，避免全局原型污染；
- 上传主路径由“文件选择器拦截”切换为更稳的“剪贴板粘贴注入”，减少 3 秒级超时损耗；
- 文件导入 UI 从目录模式切换到多文件可见导入，提升反馈。

**结果**
- 单图注入与生成链路稳定性明显提升；
- 上传时序与异常行为显著减少。

---

### 2.2 参考图未真正应用（中期）

**现象**
- 日志显示“注入成功”，但 Flow 实际未使用参考图；
- 存在“只上传了一张图”“第二条提示词使用了错误参考图”等问题。

**关键根因**
1. `chrome.runtime.sendMessage` 跨上下文传输 `ArrayBuffer` 时发生序列化损坏，导致 Blob 异常（15 bytes 类问题）；
2. 任务间参考图状态管理不清晰，导致下一任务可能继承/丢失前一任务状态。

**处理**
- 数据通道从 `ArrayBuffer` 改为 `base64` 字符串传输，再在 content script 解码回 Blob；
- 相关协议和背景脚本联动改造（`protocol.ts`、`background/index.ts`、`image-store.ts`、`execute-task.ts`）；
- 加强上传完成检测逻辑（进度出现/消失双阶段）；
- 任务执行阶段补充更细粒度日志，便于定位每张图的注入路径。

**结果**
- 参考图“有日志但无实际生效”的核心问题被修复；
- Blob 读取和注入稳定性大幅改善。

---

### 2.3 多图多提示词匹配规则补全（中后期）

**现象**
- 单提示词 + 多图片时只跑 1 个任务；
- 多提示词场景中图片分配不符合预期。

**处理（`App.svelte`）**
- 完整补齐导入匹配策略：
  - 显式文件名匹配；
  - `N 图 × 1 词`（同词扩展 N 任务）；
  - `1 图 × N 词`；
  - `M 图 × N 词`（笛卡尔积）；
  - 提示词内联文件名扫描匹配。
- 引入 `assetCache`，同名文件在队列构造时复用同一 `refId`，减少不必要重复存储。

**结果**
- 场景覆盖显著提升；
- 用户给出的“1/2/3 图 + 两条提示词”组合可正确排队执行。

---

### 2.4 重复上传效率问题与“资源面板复用”探索（后期）

**现象**
- 功能正确但效率差：同一张图在多任务中被重复上传；
- 用户明确指出：Flow 资源面板可直接选择历史图，不应重复上传。

**处理路径**
1. 先实现“已上传文件名集合 + 面板优先选择 + 失败回退上传”；
2. 从 HAR 发现关键事实：Flow 侧核心标识是 **media UUID**，不是本地文件名；
3. 将策略升级为“按 UUID 选图”：
   - 上传后从页面 `img.src` 中提取 `media.getMediaUrlRedirect?name=UUID`；
   - 记录 `filename -> mediaUuid`；
   - 后续任务尝试按 UUID 在资源面板中定位并点击，失败再回退上传。

**当前状态**
- UUID 路径代码已落地；
- 仍需继续实测确认“资源面板命中率”是否达到可接受水平（见第 6 节）。

---

### 2.5 最新阻断问题：`图片未找到 (refId...)`（刚修复）

**现象**
- 新测试中任务直接失败：`图片未找到 (refId: img_xxx)`。

**根因**
- `clearQueue/clearHistory` 会清空 IndexedDB 图片数据；
- 但 sidepanel 内存中的 `assetCache` 仍保留旧 `refId`；
- 重新导入同名文件时命中旧缓存并直接复用旧 `refId`，导致背景脚本无法从 IDB 取到 Blob。

**修复**
- `storeAsAsset` 在命中缓存时也强制 `saveImageBlob(cached.refId, imageFile)`，确保 IDB 中始终存在对应内容。

**结果**
- 该类“旧 refId 空洞”问题已从机制上修复。

## 3. 主要代码改动（按模块）

### 3.1 `src/sidepanel/App.svelte`

- 导入匹配规则升级（显式匹配 + 笛卡尔积 + 内联引用扫描）；
- 增加 `assetCache`，并在缓存命中时强制重写 IDB（最新修复）；
- 改善导入反馈与说明文案。

### 3.2 `src/shared/image-store.ts`

- 读取接口由 ArrayBuffer 路径改为 base64 安全传输路径：
  `getImageAsBase64(id)`。

### 3.3 `src/shared/protocol.ts`

- `GET_IMAGE_BLOB` 响应协议更新为 base64 字段。

### 3.4 `src/background/index.ts`

- `handleGetImageBlob` 使用 `getImageAsBase64` 返回图片数据。

### 3.5 `src/content/actions/execute-task.ts`

- 增加 Blob 内存缓存（`refId -> Blob`）；
- 参考图注入逻辑支持“先面板复用，再上传回退”；
- 新增 `filename -> mediaUuid` 维度的复用跟踪逻辑（替代纯文件名存在性判断）。

### 3.6 `src/content/actions/inject-image.ts`

- 上传成功判定与等待机制增强；
- 策略顺序优化：优先剪贴板上传；
- 新增资源面板复用逻辑：
  - 先尝试按 `mediaUuid` 在面板定位缩略图并点击；
  - 若失败，自动回退上传；
  - 上传后尝试捕获新 UUID。

## 4. 验证结论（截至当前）

### 已验证通过

- 多图 + 多提示词可按队列执行并生成；
- 参考图注入的稳定性显著高于初始版本；
- base64 通道已解决跨上下文 Blob 损坏问题；
- 最新 `refId` 丢失问题已定位并修复，构建通过。

### 待继续验证

- 资源面板按 UUID 选择的稳定命中率（不同账号/项目/列表状态下）；
- 面板命中失败时回退上传的性能与总耗时；
- 大规模批次（5~10 参考图 + 多提示词）下的复用收益。

## 5. 当前能力画像

当前版本已经从“不可用/不稳定”提升到“可跑通且可持续修复优化”的状态：

- **功能正确性**：核心链路可用，M×N 场景基本可执行；
- **稳定性**：上传与注入成功率显著改善；
- **效率**：具备“优先复用、失败回退”机制，但面板复用仍在打磨命中率；
- **可观测性**：日志信息较完整，支持继续做精准迭代。

## 6. 下一步建议（按优先级）

1. **先做真实回归**：在同一项目中连续跑 2 轮相同任务，观察第二轮是否大量命中“面板选择成功”日志；
2. **增强 UUID 捕获可靠性**：上传后增加短轮询窗口（例如 2~4s）再抓取新增 UUID，降低因页面异步渲染造成的漏捕；
3. **面板定位再收敛**：若 UUID 命中图像后点击无效，增加“可点击祖先节点”策略和二次确认；
4. **加入指标日志**：统计每轮“面板命中数/回退上传数/总注入耗时”，量化优化收益。
