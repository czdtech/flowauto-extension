## 二、架构设计


## 三、核心设计亮点

### 3.1 消息协议系统 (protocol.ts)

**定义了完整的类型安全消息协议：**

```typescript
export type AnyRequest =
  | PingRequest
  | GetPageStateRequest
  | QueueAddTasksRequest
  | ExecuteTaskRequest
  | RefMediaLookupRequest
  | ExpectDownloadRequest
  | DownloadByUrlRequest
  | ... // 共18种消息类型

export type AnyResponse =
  | PongResponse
  | PageStateResponse
  | QueueStateResponse
  | TaskResultResponse
  | ...;
```

**评价：** 消息类型通过 `typeof MSG.XXX` 关联，实现了编译时类型检查，避免了消息类型字符串的拼写错误。

### 3.2 队列状态管理 (queue-engine.ts)

**实现了完整的任务生命周期管理：**

| 方法 | 功能 |
|------|------|
| `addPrompts()` | 添加任务，自动进行 Capability Guard 检查 |
| `markTaskRunning()` / `markTaskSuccess()` / `markTaskError()` | 状态流转 |
| `retryErrors()` | 重试所有失败任务 |
| `clearHistory()` / `clearQueue()` | 清理任务，自动 GC 图片 blob |
| `appendTaskLog()` | 追加任务日志（限制每任务最多30条） |

**持久化策略：**
- 使用 `chrome.storage.local` 持久化队列状态
- 任务在 Service Worker 重启后仍能恢复

### 3.3 参考图智能复用机制 (execute-task.ts)

**三级缓存策略：**

```typescript
// 1. 内存级缓存（会话内有效）
const blobCache = new Map<string, Blob>();
const assetHashCache = new Map<string, string>();

// 2. 会话级缓存（页面刷新丢失）
const uploadedMediaByHash = new Map<string, string>();

// 3. 持久化缓存（IndexedDB，跨会话/跨任务有效）
// 存储 (projectId + assetHash) → mediaUuid 映射
```

**复用流程：**
1. 计算图片 SHA-256 哈希
2. 查询持久化映射 → 命中则按 UUID 从面板选择
3. 未命中 → 上传图片 → 捕获新的 media UUID → 回写映射

**数据增长控制：**
- TTL：超过 45 天未使用自动淘汰
- 全局上限：最多 1200 条，超出按 LRU 删除
- 项目上限：每项目最多 250 条，超出按 LRU 删除

### 3.4 兼容性守卫 (capability-guard.ts)

**自动检测并纠正不兼容的参数组合：**

```typescript
export interface CapabilityResolution {
  valid: boolean;
  corrected?: Partial<Pick<TaskItem, 'model' | 'aspectRatio'>>;
  reason?: string;
  warnings?: string[];
}
```

| 规则 | 处理 |
|------|------|
| Ingredients + 非 Veo 3.1 Fast | 自动切换到 Veo 3.1 - Fast |
| Jump To + 9:16 | 自动切换到 16:9 |
| Jump To + Veo 3.1 Fast | 自动切换到 Veo 3.1 - Quality |
| 视频模式 + 图片模型 | 标记为 skipped，提示用户切换 |

### 3.5 多策略下载 (download.ts)

**两级回退策略：**

| 策略 | 触发条件 |
|------|----------|
| 菜单点击下载 | 首选，通过 Radix UI 菜单操作 |
| URL 直接下载 | 菜单不可用时的回退，调用 `chrome.downloads.download` |

**分辨率自动降级：**
- 请求 4K 不可用时 → 尝试 2K/1080p
- 可用分辨率扫描 → 自动选择最接近的

### 3.6 生成完成检测 (generate.ts)

**多信号策略（任意一个触发即视为完成）：**

| 信号 | 检测方法 |
|------|----------|
| 信号 A — 按钮恢复 | 检测 `disabled` / `aria-disabled` 属性变化 |
| 信号 B — URL 稳定性 | 新增图片数 ≥ 期望数量，且连续 3 次轮询不变 |
| 信号 C — 无加载指示器 | 页面无进度条，且有新图片出现 |

**超时：** 图片 120 秒，视频 900 秒

### 3.7 参考图注入策略 (inject-image.ts)

**四级策略，按优先级：**

| 优先级 | 策略 | 说明 |
|--------|------|------|
| 1 | 资源面板复用 | 通过 UUID 在资源面板选择已有图片 |
| 2 | 剪贴板粘贴上传 | 构造 ClipboardEvent 触发上传 |
| 3 | 文件输入拦截 | 拦截 `<input type="file">` 的 click() 方法 |
| 4 | 拖放注入 | 构造 DragEvent 派发到提示词区域 |

### 3.8 提示词解析器 (prompt-parser.ts)

**支持多种格式：**

```typescript
// 基本格式：每行一个
一只橘猫
一只黑猫

// 自定义文件名：文件名, 提示词
cat_orange, 一只橘猫在窗台上晒太阳

// 多行提示词：空行分隔
vi_001, 一只橘猫坐在窗台上
阳光透过玻璃窗照射进来

vi_002, 一只黑猫在月光下

// 注释支持
# 这行会被忽略
cat_01, 橘猫
```

**图片导入匹配策略：**

| 场景 | 行为 |
|------|------|
| 1 图 × N 词 | 同一参考图应用于所有提示词 |
| N 图 × 1 词 | 同一提示词扩展为 N 个任务 |
| M 图 × N 词 (M≠N, both > 1) | 笛卡尔积，生成 M×N 个任务 |
| 提示词内联引用 | 扫描提示词文本中的文件名自动匹配 |

## 四、问题与风险

### 4.1 类型安全问题

**问题：** 多处使用 `as any` 类型断言

**位置：**
```typescript
// background/index.ts:60-64
const maybe = chrome.sidePanel.setPanelBehavior({
  openPanelOnActionClick: true,
} as any);
if (maybe && typeof (maybe as any).catch === "function")
  (maybe as any).catch(() => {});

// runner.ts:69
chrome.tabs.sendMessage(tabId, message as any, (response) => {
```

**影响：** 降低了 TypeScript 的类型保护，可能导致运行时错误

**建议：** 为 chrome 扩展 API 添加类型声明，或使用更严格的类型检查

### 4.2 DOM 选择器脆弱性

**问题：** 依赖 Flow 页面的特定 class 名和属性

**位置：**
```typescript
// download.ts:212
const cardContainer = btn.closest('[class*="eUdvpI"], [class*="sc-"]') || btn.parentElement?.parentElement;

// inject-image.ts:36-41
const name = normalizeForMatch(getElementName(btn));
if (name.includes("add_2") && name.includes("创建")) return btn;
```

**影响：** Flow 页面更新后可能导致自动化失败

**建议：**
- 为选择器添加版本标识
- 实现选择器外部化（配置文件）
- 添加选择器失效检测和回退逻辑

### 4.3 魔法数字问题

**问题：** 多处硬编码时间/数值

**位置：**
- `randomSleep(300, 600)` - UI 稳定等待
- `randomSleep(1500, 2500)` - 下载间隔
- `MAX_LOGS_PER_TASK = 30` - 日志上限
- `MAX_GENERATION_ATTEMPTS = 3` - 生成重试次数

**影响：** 难以维护和调整，不同环境下可能需要不同值

**建议：** 提取为配置常量，支持从 settings 动态调整

### 4.4 代码重复

**问题：** `sendMessageToTab` 函数在两个文件中重复定义

**位置：**
- `background/index.ts:117-142`
- `background/runner.ts:60-81`

**差异：** 参数默认值不同（timeoutMs）

**建议：** 提取到 `shared/` 目录作为共享工具函数

### 4.5 单文件过大

| 文件 | 行数 | 问题 |
|------|------|------|
| `App.svelte` | 1259 行 | UI 逻辑、状态管理、任务操作混合 |
| `execute-task.ts` | 490 行 | 任务编排、图片处理、复用逻辑混合 |
| `download.ts` | 372 行 | 多种下载策略混合 |

**建议：**
- `App.svelte` 拆分为子组件：`SettingsPanel`, `TaskList`, `TaskCard`
- `execute-task.ts` 将参考图处理逻辑提取到独立模块
- `download.ts` 按策略拆分为多个文件

### 4.6 日志管理问题

**问题：** 生产环境会输出大量调试日志

**位置：**
```typescript
console.log(`[FlowAuto Debug] 找到了 ${menuItems.length} 个可见的 menuitem`);
console.warn(`[FlowAuto] ⚠️ 下载菜单中没有找到请求的分辨率 (${targetRes})，正在自动降级选择...`);
```

**影响：** 调试信息泄露到生产环境，性能开销

**建议：** 引入日志分级系统（DEBUG / INFO / WARN / ERROR），生产环境关闭 DEBUG

### 4.7 潜在竞态条件

**问题：** `queue-engine.ts` 使用全局变量存储状态

```typescript
let queue: QueueState = structuredClone(DEFAULT_QUEUE_STATE);
let settings: UserSettings = structuredClone(DEFAULT_SETTINGS);
```

**影响：** 并发操作可能导致状态不一致

**建议：** 考虑引入状态锁或原子操作

### 4.8 Error Handling 不完整

**问题：** 部分错误处理为静默忽略

```typescript
if (refIds.length) void deleteImageBlobs(refIds).catch(() => {});
```

**影响：** 图片清理失败无感知，可能导致存储泄漏

**建议：** 至少记录到日志，或实现重试机制
